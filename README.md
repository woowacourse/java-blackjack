# java-blackjack

우아한테크코스 1주차 블랙잭 미션 저장소

콘솔에서 실행하는 블랙잭 게임입니다.

## 블랙잭 규칙

- **배팅**: 플레이어는 게임을 시작할 때 배팅 금액을 정해야 한다.
- **카드 점수**: 2~10은 숫자 그대로, J/Q/K는 10점, A는 1 또는 11 (버스트를 피하는 방향으로 자동 계산)
- **게임 진행**: 게임을 시작하면 플레이어는 두 장의 카드를 지급 받으며, 21을 초과하지 않으면서 21에 가깝게 만들면 이긴다. 21을 넘지 않을 경우 얼마든지 카드를 계속 뽑을 수 있다. 21을 초과할 경우 배팅 금액을 모두 잃게 된다.
- **블랙잭**: 처음 두 장의 카드 합이 21일 경우 블랙잭이 되며, 베팅 금액의 1.5배를 딜러에게 받는다. 딜러와 플레이어가 모두 동시에 블랙잭인 경우 플레이어는 베팅한 금액을 돌려받는다.
- **버스트**: 21을 초과한 경우 즉시 패배하며 배팅 금액을 모두 잃는다.
- **딜러 룰**: 딜러는 처음에 받은 2장의 합계가 16 이하면 반드시 1장의 카드를 추가로 받아야 하고, 17점 이상이면 추가로 받을 수 없다. 딜러가 21을 초과하면 그 시점까지 남아 있던 플레이어들은 승리하여 베팅 금액을 받는다.

## 주요 클래스 설명

| 클래스 | 역할 |
|--------|------|
| **GameController** | 전체 게임 흐름: 참가자 설정 → 초기 배분 → 플레이어 턴 → 딜러 턴 → 결과 출력 |
| **GameService** | 초기 배분, 카드 딜링, 승패 판정 |
| **Player** | 카드 수령, 점수 계산, 버스트/블랙잭/히트 여부 판단, 에이스 보너스 계산 |
| **CardDeck** | ShuffleStrategy로 덱 셔플 후, 카드 딜 |
| **AppConfig** | Controller, Service, View 등을 생성하고 조립 |

## 게임 플로우

1. **참가자 입력**: 쉼표로 구분된 이름 입력 (예: `pobi,jason`)
2. **초기 배분**: 참가자와 딜러에게 각각 2장씩 배분
3. **플레이어 턴**: 각 참가자는 `y`(히트) / `n`(스탠드) 입력으로 카드 추가 여부 결정
4. **딜러 턴**: 합계 16 이하면 자동 히트, 17 이상이면 스탠드
5. **결과**: 최종 카드와 점수 출력 후 승/패/무 및 딜러 전적 출력

## 구현 기능 시나리오

- **참가자 및 베팅 설정**
  - [ ] 쉼표로 구분된 문자열로 플레이어 이름 목록을 입력할 수 있다.
  - [ ] 각 플레이어별로 베팅 금액을 입력받을 수 있다.
  - [ ] 잘못된 이름/베팅 금액(음수, 0, 숫자 아님 등) 입력 시 예외/재입력을 요구한다.

- **카드 덱 및 분배**
  - [ ] 게임 시작 시 카드 덱을 셔플할 수 있다.
  - [ ] 셔플된 덱에서 플레이어와 딜러에게 각각 2장씩 초기 카드를 분배할 수 있다.
  - [ ] 덱이 비어 있는 경우 더 이상 카드를 딜할 수 없다.

- **카드 점수 및 Ace 처리**
  - [ ] J, Q, K, 10 카드는 모두 10점으로 계산한다.
  - [ ] 2~9 카드는 숫자 그대로 점수로 계산한다.
  - [ ] Ace가 존재하지 않는 경우 전체 카드 점수를 단순 합산한다.
  - [ ] Ace를 11로 계산했을 때 합계가 21을 초과하면 해당 Ace는 1로 계산한다.
  - [ ] Ace를 11로 계산했을 때 합계가 21 이하이면 해당 Ace는 11로 계산한다.
  - [ ] Ace가 여러 장일 때, 버스트를 피하는 방향으로 최적의 합을 계산한다.
  - [ ] 카드 뭉치가 비어 있는 경우 점수를 계산할 수 없다.

- **블랙잭 / 버스트 판정**
  - [ ] 처음 받은 두 장의 카드 합이 21이면 블랙잭으로 판정한다.
  - [ ] 플레이어와 딜러가 동시에 블랙잭인 경우 무승부로 처리한다.
  - [ ] 카드 합이 21을 초과하면 즉시 버스트로 판정한다.

- **플레이어/딜러 턴 진행**
  - [ ] 플레이어는 `y`(히트)를 입력하면 카드를 한 장 더 받을 수 있다.
  - [ ] 플레이어는 `n`(스탠드)을 입력하면 더 이상 카드를 받지 않는다.
  - [ ] 히트/스탠드 입력이 잘못된 경우 재입력을 요구한다.
  - [ ] 딜러의 카드 합이 16 이하인 경우 반드시 한 장을 더 받는다.
  - [ ] 딜러의 카드 합이 17 이상인 경우 더 이상 카드를 받지 않는다.

- **승패 및 수익 계산**
  - [ ] 플레이어와 딜러의 최종 점수를 비교하여 승/패/무를 판정한다.
  - [ ] 플레이어가 블랙잭으로 승리한 경우 베팅 금액의 1.5배를 수익으로 계산한다.
  - [ ] 일반 승리/패배/무승부에 따라 각 플레이어의 수익/손실을 계산한다.
  - [ ] 모든 플레이어의 수익/손실을 기준으로 딜러의 최종 수익을 계산한다.

- **최종 출력**
  - [ ] 각 플레이어와 딜러의 최종 카드 목록과 점수를 출력한다.
  - [ ] 각 플레이어의 승/패/무 결과와 수익/손실 금액을 출력한다.
  - [ ] 딜러의 최종 수익을 요약하여 출력한다.

## 사이클2 - 미션 중 기록

### 기능 추가로 인해 수정된 위치 개수

20개 파일을 변경했다. 베팅 룰 추가로 인한 BettingRule 인터페이스와 기본 DefaultBettingRule, 사용자의 최종 이익을 저장하기 위한 UserProfit 레코드 생성과 이로 인한 테스트 생성 등으로 여러 파일들이 생겼으며, 달라진 입출력을 위해 컨트롤러나 서비스 등 여러 파일들을 수정하게 되었다. 사이클1의 마지막 커밋 기준 추가된 라인은 290줄, 삭제된 라인은 20줄이다.

### 사이클1 때보다 수정 범위가 줄었는가/늘었는가

생각보다 수정 범위가 적었다고 생각한다. 그 이유는, 기존 사이클1에서 리팩토링을 통해 객체 간의 책임을 보다 명확히 했기 때문이라고 생각한다. 우선 베팅 룰을 설계하는 과정에서, ShuffleStrategy처럼 인터페이스 전략으로 여러 전략들에 유연하게 대체하기로 결정했기 때문에 새로운 인터페이스만 테스트하면 되었다. 또한 처음에는 User가 Profit을 가지고 있게 하기 위해 User 클래스에 profit 변수를 추가하려 했지만, 클래스 변수가 세개를 넘으면 안되는 요구사항을 만족하기 위해 판당 profit을 저장해놓는 UserProfit 레코드를 따로 구현하였다. 이렇게 책임을 분리해놨기 때문에 수익 계산과 관련없는 부분은 아예 건드릴 필요가 없었다.

BettingRule과 UserProfit을 구현한 뒤에는, 반환 타입이나 컨트롤러에서 입출력 정도만 수정하면 되는 거여서 사이클1보다는 수정이 훨씬 준 것 같다.

```
## 개선 방법
- 순수 함수로 분리
- 의존성 주입
- 작은 단위 객체 설계
```

를 통해 UserProfit이라는 객체를 설계했다. 처음에는 GameResult 안에 profit 필드를 두고, GameResult에서 수익을 계산하는 방식으로 구현하려 했다. 하지만 이렇게 하면 GameResult가 승패 결과와 수익 계산이라는 서로 다른 책임을 갖게 되어 테스트가 어려워졌다. 순수 함수로 분리하고 작은 단위 객체를 설계하라는 원칙에 따라, GameResult는 승패 판정만 담당하고 수익 계산 결과를 담는 UserProfit 레코드를 따로 두기로 했다. 이렇게 하니 수익 로직을 독립적으로 검증할 수 있어 테스트 작성이 훨씬 수월해졌다.

### 테스트가 설계를 도운 순간

```
1.2 테스트 어려움에 대한 대응
설계 수정: 테스트 코드를 작성하기 어렵다면, 그것은 코드의 설계에 문제가 있다는 신호입니다.

대응 방법: 메서드를 더 작게 분리하거나 의존성을 분리하여 테스트하기 쉬운 설계로 개선합니다.
```

를 통해 createUsersProfit, createEachUserProfit, upsertDealerProfit 메소드를 분리해 보다 테스트가 용이하도록 수정했다. 이렇게 하니 가독성도 좋아져서 만족스럽다.
