# java-blackjack

## 기능 요구 사항

### 기능

- [x] 플레이어는 2명 이상, 5명 이하여야 한다.
- [x] 카드 덱은 총 52장으로 이루어져 있다.
    - 카드 종류는 스페이드, 하트, 클로버, 다이아몬드가 있다.
    - 숫자는 2부터 10까지 있고, 문자는 A,J,Q,K가 있다.
- [x] 카드 덱의 순서를 무작위로 섞는다.
- [x] Ace는 1 또는 11로 계산할 수 있다.
- [x] King, Queen, Jack은 각각 10으로 계산한다.
- [x] 플레이어와 딜러는 무작위 카드 2장을 받는다.
- [x] 플레이어는 21이 넘지 않으면 얼마든지 카드를 계속 뽑을 수 있다.
- [x] 딜러는 2장의 합계가 16점 이하면 반드시 1장을 카드를 추가로 받는다.
- [x] 딜러는 2장의 합계가 17점 이상이면 추가로 받을 수 없다.
- [x] 카드의 숫자 합을 계산한다.
- [x] 합이 21이 넘을 경우 버스트로 판정한다.
    - [x] 딜러가 버스트일 경우 플레이어가 승리한다.
    - [x] 플레이어가 모두 버스트일 경우 딜러가 승리한다.
    - [x] 플레이어 일부가 버스트가 아닐 경우, 버스트인 플레이어만 패배로 판정한다.
- [x] 카드를 추가로 뽑아 21을 초과할 경우 배팅 금액을 모두 잃게 된다.
- [x] 처음 두 장의 카드 합이 21일 경우 블랙잭이 된다.
- [x] 블랙잭일 경우 배팅 금액의 1.5 배를 딜러에게 받는다.
- [x] 딜러와 플레이어가 모두 동시에 블랙잭인 경우 플레이어는 배팅한 금액을 돌려받는다.
- [x] 딜러가 21을 초과하면 그 시점까지 남아 있던 플레이어들은 가지고 있는 패에 상관 없이 승리해 베팅 금액을 받는다.

### 입력

- [x] 참여할 플레이어의 이름을 입력한다.
    - [x] 쉼표 기준으로 분리한다.
    - [x] 이름 길이는 최소 2자 이상, 5자 이하로 입력한다.
  ```
    게임에 참여할 사람의 이름을 입력하세요.(쉼표 기준으로 분리)
    pobi,jason
  ```
- [x] 각 플레이어에게 카드를 더 받을지 입력 받는다.
    ```
    pobi는 한장의 카드를 더 받겠습니까?(예는 y, 아니오는 n)
    y
    ```
- [x] 각 플레이어에게 배팅 금액을 입력 받는다.
  ```
  pobi의 배팅 금액은?
  10000
  ```

### 출력

- [x] 첫 시작 카드 현황을 출력한다.
    - 딜러의 카드는 한장만 출력하고 플레이어의 카드는 모두 출력한다.
      ```
      딜러와 pobi, jason에게 2장을 나누었습니다.
      딜러카드: 3다이아몬드
      pobi카드: 2하트, 8스페이드
      jason카드: 7클로버, K스페이드
      ```
- [x] 플레이어의 카드 현황을 출력한다.
  ```
  pobi카드: 2하트, 8스페이드, A클로버
  ```
- [x] 딜러의 카드 추가 발급 여부를 출력한다.
    - 딜러의 카드가 16 이하라면 다음과 같은 메시지를 출력한다.
      ```
      딜러는 16이하라 한장의 카드를 더 받았습니다.
      ```
- [x] 딜러와 플레이어의 카드 현황 및 점수와 최종 수익을 출력한다.
  ```
  딜러카드: 3다이아몬드, 9클로버, 8다이아몬드 - 결과: 20
  pobi카드: 2하트, 8스페이드, A클로버 - 결과: 21
  jason카드: 7클로버, K스페이드 - 결과: 17
    
  ## 최종 수익
  딜러: 10000
  pobi: 10000
  jason: -20000
  ```

## **프로그래밍 요구 사항**

- 자바 코드 컨벤션을 지키면서 프로그래밍한다.
    - 기본적으로[Java Style Guide](https://github.com/woowacourse/woowacourse-docs/tree/master/styleguide/java)을 원칙으로 한다.
- indent(인덴트, 들여쓰기) depth를 2를 넘지 않도록 구현한다. 1까지만 허용한다.
    - 예를 들어 while문 안에 if문이 있으면 들여쓰기는 2이다.
    - 힌트: indent(인덴트, 들여쓰기) depth를 줄이는 좋은 방법은 함수(또는 메서드)를 분리하면 된다.
- 3항 연산자를 쓰지 않는다.
- else 예약어를 쓰지 않는다.
    - else 예약어를 쓰지 말라고 하니 switch/case로 구현하는 경우가 있는데 switch/case도 허용하지 않는다.
    - 힌트: if문에서 값을 반환하는 방식으로 구현하면 else 예약어를 사용하지 않아도 된다.
- 모든 기능을 TDD로 구현해 단위 테스트가 존재해야 한다. 단, UI(System.out, System.in) 로직은 제외
    - 핵심 로직을 구현하는 코드와 UI를 담당하는 로직을 구분한다.
    - UI 로직을 InputView, ResultView와 같은 클래스를 추가해 분리한다.
- 함수(또는 메서드)의 길이가 10라인을 넘어가지 않도록 구현한다.
    - 함수(또는 메소드)가 한 가지 일만 하도록 최대한 작게 만들어라.
- 배열 대신 컬렉션을 사용한다.
- 모든 원시 값과 문자열을 포장한다.
- 줄여 쓰지 않는다(축약 금지).
- 일급 컬렉션을 쓴다.

### **추가된 요구 사항**

- 모든 엔티티를 작게 유지한다.
- 3개 이상의 인스턴스 변수를 가진 클래스를 쓰지 않는다.
- 딜러와 플레이어에서 발생하는 중복 코드를 제거해야 한다.

## 페어와 정한 우리들만의 규칙

```
1. 책임 분리 기준
(If-Then) 만약 메서드 안에서 데이터와 로직의 거리가 멀게 느껴지거나, 메서드의 이름을 직관적으로 짓기 어렵다면
→ 객체의 책임이 혼재되어 있다는 강한 신호이므로, 로직을 데이터가 있는 쪽으로 옮기거나 책임을 분리한다.
(If-Then) 만약 한 클래스 안에 상태(필드)나 행위(메서드)가 너무 많아져 절차지향적인 흐름이 보인다면
→ "이 정보/행위를 가장 잘 알고 스스로 판단할 수 있는 주체는 누구인가?"를 질문하여 책임을 재분배한다.

(기준) 책임 분리 기준: 객체가 본인의 상태만 사용하는가

2. 변경 범위 제한 규칙
(If-Then) 만약 새로운 요구사항을 반영하기 위해 기존에 잘 돌아가던 코드를 직접 수정하고 있다면
→ 개방-폐쇄 원칙을 위반한 것이므로, 기존 코드 수정 대신 새로운 클래스나 메서드를 추가하는 방식으로 확장이 가능하도록 재설계한다.

(기준) 만약 기능 하나를 추가하거나 수정했을 때, 5개의 이상의 메서드가 수정이 필요하면 무조건 재설계하고, 4개 이하의 메서드가 수정이 필요하면 재설계를 고려한다.

3. 테스트와 설계의 관계 규칙
(If-Then) 만약 내부 구현 로직을 명확히 수정했는데도 테스트가 전혀 깨지지 않거나, 반대로 예상치 못한 엉뚱한 곳에서 테스트가 깨진다면
→ 설계된 의도와 실제 코드의 의존성이 다르게 얽혀 있다는 뜻이므로, 도메인 간의 불필요한 결합이 없는지 점검한다.
(If-Then) 만약 새로운 상태를 도입했을 때, 이와 무관한 기존 도메인의 테스트가 깨진다면
→ 책임 분리가 제대로 되지 않아 사이드 이펙트가 발생한 것이므로, 각 객체가 독립적으로 동작하고 테스트될 수 있도록 격리한다.

(기준) 테스트 실패의 양면성: 테스트가 깨지는 것 자체가 무조건 나쁜 것은 아니다. 내가 의도한 로직 수정으로 인해 정확한 위치의 테스트가 실패했다면, 
이는 프로그램의 변화를 올바르게 인지시켜 주는 '좋은 지표'가 된다. 반면, 무관한 테스트가 깨진다면 '설계의 문제'로 받아들여야 한다.
```
