#### 기능 추가로 수정한 위치 개수

사이클 2 시작 기점으로 봤을 때 기존 파일은 20개 수정되었다.

주요 수정 지점:

- `BlackjackResult`: 내부 로직을 `MatchJudge`, `DealerMatchCount`, `ProfitMap`으로 위임
- `BlackjackController`: 베팅 흐름 추가 및 메서드 순서 재배치 (+44/-41)
- `BlackjackResultDto`, `PlayerResultDto`: 승패 카운트 → 수익 금액 구조로 변경
- `OutputView`: 출력 포맷 승패 → 수익으로 변경
- 예외 throw 문: `IllegalArgumentException`, `IllegalStateException` → `BlackjackException` 계열로 통일

#### 사이클 1에 비해 수정 범위

> **수정 자체는 사이클 2에서 늘었다.**

사이클1은 빈 프로젝트에서 시작이라 기능 추가 시 기존 파일 수정이 거의 없었음(1개). 따라서 사이클1 리팩터링 구간(0190b35 → 369a2dff)과 사이클2 기능 추가 구간(369a2dff → HEAD)을
비교하였다.

|       | 사이클1 리팩터링       | 사이클2 기능 추가      |
|-------|-----------------|-----------------|
| 신규 파일 | 13개 (+379)      | 13개 (+281)      |
| 수정 파일 | 11개 (+164/-186) | 20개 (+356/-160) |
| 수정 성격 | 삭제 > 추가 (갈아엎기)  | 추가 > 삭제 (확장)    |

수정 파일 수는 사이클2가 약 2배(11 → 20)로 늘었다. 다만 수정 성격이 다르다:

- **사이클1**: 기존 메서드의 이름, 시그니처를 변경하는 **교체** 위주. 호출처까지 연쇄 수정 발생.
- **사이클2**: 기존 메서드의 시그니처는 유지하고 내부 구현만 변경하는 **확장** 위주. 실제 로직은 신규 클래스(`Bet`, `MatchJudge`, `ProfitMap` 등)에 추가.

수정 범위는 늘었지만 확장 위주로 늘어난 것이라 긍정적이다. 사이클1 리팩터링에서 구조를 잡아놓은(`Cards → Deck/Hand` 분리, DTO 도입, 패키지 재구성 등) 덕분에 사이클2에서 기존 코드를 뜯지
않고 새 클래스 추가로 기능을 확장할 수 있었다.

#### 규칙 적용으로 변경한 코드

##### 1. private 메서드 -> 별도 객체 분리

> private method가 모여 한 책임을 구성할 수 있다면 -> 바로 객체로 빼는 방법 고려

- BlackjackResult에서 판정 로직을 유연하게 처리하기 위해 관련 메소드 (`isPlayerLose`, `isPlayerScoreEqualsDealer`) 를`MatchJudge`로 분리하였다.

##### 2. 테스트 중복 로직 -> test 전용 유틸로 분리

> 테스트 과정에서 중복되는 로직이 테스트 전체에 걸쳐 있으면 -> test 전용 유틸로 뺀다.

`BetTestUtil`로 `createSinglePlayerSet()`, `createMultiPlayerSet()`을 통해 BetMap 생성 코드 중복을 줄였다.

##### 3. 상수화

> 하드코딩된 값이 보인다면 -> 상수화한다

- 21 → GameConstant.BLACKJACK_SCORE
- 2 → GameConstant.NUM_OF_TO_BLACKJACK
- 1 → NumberRangeValidator.POSITIVE_MIN

##### 4. 생성자 2개 이상 -> 정적 팩터리 메서드

> 생성자가 2개라면 오버로딩 활용, 3개 이상이면 정적 팩터리 메서드 적용한다.

- `NumberRangeValidator`
    - 기본 생성자, 범위 지정 생성자 2개를 작성했다.
    - 추후 양수 범위만 간단하게 뽑아내고자 `createPositiveRange()` 정적 팩터리 메서드 추가

##### 5. DTO 기반 테스트

> 테스트가 어려운 도메인일 때 DTO로 변환될 수 있으면 -> DTO 기반으로 테스트한다

`BlackjackResult` 테스트에서 BlackjackResult 인스턴스를 직접 검증하지 않고 `BlackjackResultDto`를 통해 딜러 수익, 플레이어 수익을 검증했다.

#### 테스트가 설계를 도운 순간

##### 1. Bet VO 타입 변경

참가자의 베팅 금액 유효성을 확인하기 위해 `Bet` VO 클래스를 테스트하던 중 "참가자가 `int` 범위를 넘는 값을 넣을 때"를 고려하지 않았음을 깨달았다.

1. VO에 담을 타입을 `long`으로 바꿨다.
2. 입력을 단순히 수로 되어 있는지 검증하는게 아니라 수가 특정 범위에 포함되는지 판단하는 validator를 기반으로 검증하는 방식으로 개편했다.

##### 2. `MatchJudge`의 판정 순서가 보장 안되는 문제

`MatchJudge` 클래스의 딜러와 참가자의 승패를 판정하는 로직을 각 승패결과(`MatchResult`)에 대응되는 Predicate 판정 메소드로 구현했다.

이때 Map 생성을 `Map.of()`로 했는데, 이렇게 되면 판정 순서가 보장되지 않아 의도치 않은 버그가 발생할 수 있었다. 예를 들어, 블랙잭으로 이겼는데도 일반적인 이김 판정으로 처리될 수 있었다. 이를
`LinkedHashMap`으로 변경하여 판정 순서를 보장하였다.
