## 사이클 질문
- 기능 추가 시 왜 기존 코드가 깨지는가?
- 책임 분리의 기준은 무엇인가?
- 테스트는 설계를 어떻게 돕는가?
- 변경 범위를 최소화하는 구조란?

### 배팅 기능

1. 수정이 필요한 파일 개수 - controller, service, participant, 
2. 수정이 필요한 메서드 개수 - controller(1)
3. 가장 많이 수정해야 하는 클래스 controller(run(), batting()), service(batting)
4. 수정시 깨질 것 같은 테스트 개수

### 질문
- 한 곳만 수정해도 되는 기능이 있는가
- 여러곳을 수정해야 하는 기능은 무엇인가
- 왜 여러곳을 수정해야하는가

### 넓은 변경 범위의 기준
□ 이 코드 고치면, 다른 곳도 같이 고쳐야 하나?
□ 고치려는 곳까지 가는데 관계없는 코드를 읽어야 하나?
□ 새 기능이 기존 코드 블록 안으로 끼어드나?
□ 이 클래스/메서드가 바뀌는 이유가 두 가지 이상인가 ?

### 추가된 규칙
1. 하나의 클래스는 하나의 책임만 가져야 한다.(변경의 이유가 서로 다른 기능들이 한 클래스에 있다면 즉시 분리)
2. 기능 수정 시 영향을 받는 클래스를 최소화한다. -> 생각해봤는데 단순 클래스 변경은 흐름상 엮여있으면 상관없고 설계 문제가 있을 때(캡슐화 x, 책임 분배 잘못, 순환 의존성)
3. 외부에서 주입 받는 방식으로 개선
